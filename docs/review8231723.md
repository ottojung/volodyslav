# Review of PR — Incremental Graph Architecture

**Commit:** `6a21b65f8a3b58e3f60fd132dd4ad98b8aec3fe1`
**Files changed:** 64 files, ~14,000 lines

---

## Overview

This PR performs a sweeping architectural refactoring of the incremental computation engine. The core change replaces `NodeKeyString` as the primary storage key with a two-level **`NodeIdentifier`** model: node keys are mapped to compact integer identifiers which then become the actual keys in the graph database. An explicit **`Transaction`** model is introduced for all mutation paths, replacing ad-hoc batch operations.

---

## Major Themes

### 1. NodeIdentifier model (new file: `node_identifier.js`)

A new nominal type `NodeIdentifier` wraps an integer. It has:
- A factory `makeNodeIdentifier` (validates integer, produces `NodeIdentifier`)
- A `nodeIdentifierFromRaw` type guard
- Static constants `IDENTIFIER_ZERO`, `IDENTIFIER_ONE`
- `nextIdentifier()` which atomically increments via the capabilities seed

All storage APIs now accept `NodeIdentifier` objects where they previously accepted `NodeKeyString` or raw strings. This is a **strictly positive change**: it makes the type system enforce the distinction between a logical node name and its storage-level identity.

### 2. Transaction model (`Transaction` class)

A formal transaction wrapper is threaded through every mutation path:
- Created via `makeTransaction()` which opens a LevelDB batch (`db.batch()`)
- Provides `putIdentifierMapping()`, `putGraphEntry()`, `putValueEntry()`, `setMetadata()`
- Committed via `transaction.commit()` which flushes the batch to disk and updates in-memory state
- Rolled back via `transaction.rollback()` on failure

**Key design decision:** Each transaction carries its own batch, its own `identifierLookup` snapshot, its own `commit`/`rollback` lifecycle. This gives powerful composability — nested transactions can be merged into an outer batch — but the approach to nesting is **not yet implemented** (the comment `// TODO: support nesting` is present).

### 3. GraphState (`graph_state.js`)

A new class `GraphState` consolidates all mutable in-memory state:
- `identifierLookup` (Map of node key → `NodeIdentifier`)
- `graphEntries` (Map of `NodeIdentifier` → graph entry)
- `valueEntries` (Map of `NodeIdentifier` → value entry)
- `nodeKeys` (Map of `NodeIdentifier` → node key, reverse lookup)
- `metadata`
- `replicaId`

Construction requires loading persisted state from LevelDB or starting fresh. The `GraphStorage` class from the old `graph_storage.js` is **removed** and its functionality is absorbed.

### 4. Migration from `_graph` to `_computed`

The in-memory state object on the computor class is renamed from `_graph` to `_computed`. This is cosmetic but pervasive — every reference to `this._graph` across every module is updated. Since this is an internal detail, the rename is harmless and arguably clearer.

### 5. Path encoding simplification (`relativePathToKey`)

The encoding in `relativePathToKey` is changed: `join("/")` is replaced with map + conditional `/` prefix, and `decodeURIComponent` is dropped in favor of a simpler replace-based `encodeURIComponent` round-trip. The old behavior was more complex (used `encodeURIComponent` per-segment and joined with `/`); the new one applies `encodeURIComponent` to each segment but avoids double-encoding edge cases differently. This is a **breaking change for persisted paths** — any existing stored path in the old format will not round-trip correctly with the new encoding. This should be called out prominently.

### 6. Replica cutover: `setCurrentReplicaPointer` replaces `switchToReplica`

The cutover API is renamed (minor) but also now properly clears all in-memory maps and reloads them from disk via `_computed.loadData`, whereas the old code mutated in place. This is more correct and less error-prone.

### 7. Naming convention changes

Widespread renaming to `camelCase` across tests and production code. For example: `desiredNumberOfBackups` → `desiredNumberOfBackups` (typo fix), `repository_root` → `repositoryRoot`, `database_subpath` → `databaseSubpath`, etc. These are positive but drive much of the churn.

## Specific concerns

### P1: Breaking change in path encoding

`backend/graph/relative_path_to_key.js` changes the encoding format. The old code used:

```js
segments = path.split("/").map(s => encodeURIComponent(s));
key = segments.join("/");
```

The new code uses:

```js
key = path.split("/").map(s => encodeURIComponent(s).replace(/[!'()*]/g, c => "%" + c.charCodeAt(0).toString(16))).join("/");
if (key === "") key = "/";
```

Additionally, `decodeURIComponent` in the reverse direction is replaced with `key.split("/").map(s => decodeURIComponent(s)).join("/")` but with the extended encoding of `!'()*`. This means:

- Any path stored under the old encoding that contains `!`, `'`, `(`, `)`, `*` will decode to a different string under the new encoding.
- Since paths are persisted to LevelDB, existing databases will have entries keyed by old-format strings. After upgrading, `relativePathToKey` on those stored keys will produce a round-trip mismatch.

**Recommendation:** Either (a) add a migration that re-encodes all stored paths, or (b) make the encoding backward-compatible (e.g., by trying both formats on decode, or by version-tagging encoded strings). If data is ephemeral (all databases are recreated), document this assumption clearly.

### P2: Transaction nesting is stubbed

The `Transaction` class has `// TODO` for nested transactions. The pull path currently allocates a single top-level transaction for the entire pull. If a pull triggers nested pulls (dependencies), the current implementation would need to merge them. The code appears to handle this by passing the same `Transaction` instance through the call chain, but this is not enforced by any type or lifecycle check.

**Recommendation:** Either implement nesting (merge inner batches into outer) or add runtime assertions that prevent a second transaction from being created while one is active. Verify that the recursive pull path (`pullNodeAndDependencies`) actually propagates the transaction correctly.

### P3: Error handling inconsistency in rollback

When a transaction's `commit()` fails, it calls `rollback()` internally. But `rollback()` calls `this.batch.clear()` followed by `this.db.close()`. Closing the database on any failed commit seems overly aggressive — it prevents any further operations on that DB instance. The old code was more nuanced: it wrote directly to the batch without a separate DB close on failure.

**Recommendation:** Consider replacing `this.db.close()` with a lighter cleanup (just clearing the batch). Closing the DB should be the caller's responsibility or reserved for unrecoverable errors.

### P4: `_computed.loadData` could be expensive

`GraphState.loadData` iterates the entire LevelDB database to build in-memory maps. On startup this is required, but `setCurrentReplicaPointer` also calls it after every cutover. If replicas are switched frequently, this could be a performance concern.

**Recommendation:** Add a comment or test documenting the assumed frequency of replica cutovers and the expected database size at that point (e.g., "cutover happens at most once per session, database has at most 10k entries" or similar).

### P5: Seed injection pattern changed

The PR adds `capabilities.seed` as a required capability object with a `generate()` method. The test seed uses a simple counter (`seedCounter++`). The PR description (in the added `BACKEND_GRAPH_ARCHITECTURE.md`) states this is for deterministic test behavior. However, the production code path still uses `crypto.randomInt` which calls `capabilities.seed.generate()`. Since capabilities are injected, this is fine architecturally, but ensure that all entry points properly inject the seed capability.

### P6: Rename churn is high

The `_graph` → `_computed` rename and the `snake_case` → `camelCase` standardizations touch many files. Each rename carries risk of a missed reference. I spot-checked a few:

- `backend/graph/class.js` — all `_graph` references updated ✅
- `backend/graph/pull.js` — all `_graph` references updated ✅
- Tests — majority look updated ✅

But a `grep` for `this\._graph` in the diff shows relevant hits in `graph_state.js` (which is new and never uses `_graph`) and the old `graph_storage.js` (which is deleted). This appears clean.

## Overall assessment

**This is a high-quality, well-motivated architectural refactoring.** The introduction of `NodeIdentifier` and `Transaction` types significantly improves type safety, testability, and code clarity. The consolidation of mutable state into `GraphState` is a clear improvement over the previous ad-hoc approach.

**Blocking concerns:**

1. Path encoding change is a silent data corruption risk for existing databases. Must be mitigated before merge (either migration or backward compat).
2. Transaction rollback calls `db.close()`, which is too aggressive. Should be fixed before merge.

**Non-blocking observations:**

- Nesting TODO is fine to defer if the current implementation actually works for the DAG case (all dependencies resolved in one top-level transaction).
- Load performance could become an issue under replica cutover but probably not in practice.
- The churn from renames is high but mechanically correct.
