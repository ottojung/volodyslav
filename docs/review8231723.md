## 0. What this review covers

## 1. Transaction isolation model — per-call instead of shared batch

**Severity: moderate (intentional but notable)**

**What changed:** Each `pullNode` call now creates and commits its *own* independent `Transaction`. Previously, an entire pull tree (node → dependencies → transitives) shared a single LevelDB batch; either everything committed atomically or nothing did.

In the new code (`pull.js:51`):
```
pullNode(A)
  → withTransaction(TX1)
    → resolve A, discover dependency B
    → pullNode(B)
      → withTransaction(TX2)     ← commits independently
      → TX2 committed, B visible on disk
    → continue TX1 using B's committed result
  → TX1 commits
```

**Consequence:** If pull(A) → depends on B → depends on C, and C's pull succeeds but B's computation fails (throws), C's writes are already persisted — there is no rollback. The old atomicity guarantee across a dependency tree is gone.

**Also affects concurrency visibility:** With the old shared-batch model, no intermediate state was visible until the top-level pull completed. Now, each nested pull's writes are committed to LevelDB before the parent proceeds. A concurrent observer polling the database could see partial results.

**Context:** For this application (personal event logging, non-adversarial client, single-user) this is unlikely to cause problems in practice. Dependency trees are shallow, concurrent operations are rare, and partial commits of dependency computations are better than state loss. But the change in semantics should be documented — it's a deliberate design choice, not an oversight.

## 2. Path encoding — snapshot directory format breakage

**Severity: high for snapshot users; none for database users**

**`encoding.js`** rewrote both `keyToRelativePath` and `relativePathToKey`:
- Old: `namespace/sublevel/head/arg1/arg2...` (multi-segment, parsed NodeKey JSON)
- New: `namespace/sublevel/opaqueString` (single encoded segment)

**Old-format snapshot directories cannot be read by the new code.** Every data-sublevel path produces a different raw key. For example:

| Raw key (old) | Old path | New `relativePathToKey` reads it as |
|---|---|---|
| `!x!!values!{"head":"event","args":["abc"]}` | `x/values/event/abc` | `!x!!values!event/abc` ← completely different |

**The fix exists:** A migration pipeline (`migration_runner.js`, `legacyStringToNodeIdentifier`) converts old databases to the new identifier-based format at the LevelDB level, not at the snapshot-directory level. Snapshot *filesystem* dumps produced by the old `keyToRelativePath` would need to be regenerated after migration — there is no code to read old-format snapshots.

**Likely acceptable** if snapshots are short-lived artifacts always regenerated from the database. If old snapshot directories are expected to be importable indefinitely, this needs addressing.

## 3. Error handling asymmetry in `withTransaction`

**Severity: low (theoretical only)**

In `graph_state.js:4612–4688`, the commit sequence is:

```
1. activeSchemaStorage.batch(operations)    ← LevelDB flush (can fail)
2. commitTransactionLookup(tx.identifierLookup)  ← apply overlay to in-memory lookup
```

Step 1 writes to disk. Step 2 updates volatile state. They are in this order intentionally (disk-first invariant). But there is no handling for the case where step 1 succeeds and step 2 throws. (Step 2 iterates two Maps and sets entries — it should never throw under normal conditions, but e.g. an out-of-memory allocation failure in V8 could.)

If step 2 throws:
- Data is safely on disk.
- The in-memory `identifierLookup` is *missing* the new entries.
- Subsequent reads from `getActiveIdentifierLookup()` will not find the identifiers.
- Recovery requires re-opening the database (which reloads from disk).

**Minimal fix:** Either wrap step 2 in a try-catch that triggers a full reload of `_computed` from disk, or add a JSDoc `@throws` documenting that the in-memory state may be stale after a partial failure.

## 4. Identifier allocation during invalidation

**Severity: low (probably fine, but notable)**

In `invalidate.js`, when the invalidation path encounters input keys that haven't been allocated yet, it allocates new identifiers for them via `getOrAllocateNodeIdentifier`:

```js
for (const inputKey of nodeDefinition.inputKeys) {
    const existing = lookupNodeIdentifier(tx, inputKey);
    if (existing !== undefined) {
        staticInputIdentifiers.push(existing);
    } else {
        staticInputIdentifiers.push(
            getOrAllocateNodeIdentifier(tx, incrementalGraph.rootDatabase, inputKey)
        );
    }
}
```

This means: **invalidating a node can create identifier entries for keys that have never been pulled.** The key will now have a reserved identifier even if it's never computed.

This is new behavior — the old invalidation code never allocated anything. The identifiers are valid (they map correctly), but the unused identifier entries persist in the database. If the space of node keys is bounded, this is harmless.

## 5. Early committed-storage freshness check

**Severity: low (semantically correct)**

`pull.js:40–47` checks the *committed* storage for freshness before opening a Transaction:

```js
const committedIdentifier = graph.lookupNodeIdentifier(nodeKeyStr);
if (committedIdentifier !== undefined) {
    const nodeFreshness = await graph.storage.freshness.get(committedIdentifier);
    if (nodeFreshness === "up-to-date") {
        const result = await graph.storage.values.get(committedIdentifier);
        if (result !== undefined) {
            return { value: result, status: "cached" };
        }
    }
}
```

This means an up-to-date node in committed storage returns immediately, even if there's an in-flight transaction that has updated the node but not yet committed. The caller sees the pre-transaction value — which is correct for a read committed isolation level.

**No correctness concern**, but differs from the old behavior where everything went through the shared batch and the freshest value (including uncommitted writes) was visible inside the batch. This is a shift from "read uncommitted within batch" to "read committed across transactions."

## 6. Dropped "Impossible" error for missing node value

**Severity: low (behavioral change)**

Old code in `maybeRecalculate` threw:
```js
throw new Error("Impossible: up-to-date node has no stored value");
```

New code silently falls through to recomputation. This changes failure behavior: a corrupted database (node marked up-to-date but value missing) will silently recompute instead of crashing. Whether this is an improvement depends on preference; it's a change worth noting.

## 7. Migration layer (`legacyStringToNodeIdentifier`)

**Severity: needs manual verification**

The migration helper `legacyStringToNodeIdentifier` in `migration_storage.js` accepts three formats:

1. Modern 9-letter identifiers (`/^[a-z]{9}$/`)
2. Legacy serialized NodeKey JSON (`{"head":"...","args":[...]}`)
3. Legacy bare node names (`all_events`, `user_alice`, etc.)

This must correctly convert all old-format keys in the database:
- `inputs` records (arrays of NodeKey strings)
- `revdeps` arrays (arrays of NodeKey strings)
- `values`, `freshness`, `counters`, `timestamps` keys
- The `identifiers_keys_map` global key (which didn't exist in old format, so a fresh empty one is created)

I did not audit each migration path exhaustively, but the three-format detection looks correct for the common cases.

## Summary

| Issue | Severity | Verdict |
|-------|----------|---------|
| 1. Per-call Transactions break atomicity across dependency trees | Moderate | Intentional design choice; document |
| 2. Snapshot directory format incompatible with old code | High | Migration exists at DB level; snapshots must be regenerated |
| 3. In-memory state may trail disk after commitTransactionLookup failure | Low | Add guard or document |
| 4. Invalidation allocates identifiers for unpulled keys | Low | Harmless; note in docs |
| 5. Early freshness check reads committed, not in-flight state | Low | Read-committed semantics; correct |
| 6. Dropped "Impossible" error for missing value | Low | Behavioral change; acceptable |
| 7. `legacyStringToNodeIdentifier` coverage | Needs verification | Spot-checked; appears correct |

**Overall:** The architecture is sound. The transaction isolation change (item 1) is the most consequential semantic shift — it trades atomic dependency-tree commits for simpler per-node transaction boundaries. Everything else is either well-handled (migration for encoding) or low-risk.
