## Overview

This is a foundational architectural refactor. It replaces the old path-based/semantic-key node addressing scheme with **deterministic fingerprint-index node identifiers**, introduces a **transaction-based volatile-consistency model** with overlay-based identifier lookups, and replaces the monolithic locking with a **fine-grained "cosmic observatory"** lock hierarchy. The sync merge is decomposed from one monolithic module into five focused sub-modules.

The central change: nodes are no longer stored under semantic keys like `all_events` in LevelDB. Every node gets an opaque-but-deterministic identifier like `1-abcdefghi` (base36-index + machine fingerprint), and a persisted `identifiers_keys_map` maintains the bijection.

## 1. Architectural Changes

### 1.1 Deterministic Node Identifiers (`database/node_identifier.js`)

**Format:** `<base36-index>-<fingerprint>` (e.g., `1-abcdefghi`, `z-abcdefghi`, `10-abcdefghi`)

- The index is a **monotonic allocation counter** starting at 1, local to each host.
- The fingerprint is a **random lowercase-alpha string** (9+ chars), generated once per database at first boot and persisted in `r/global/fingerprint`.
- The combined identifier is **globally unique across hosts** (via the fingerprint suffix) and **deterministic** (same index always refers to the same node on a given host).

**Design rationale:** Identifiers are validated at lifecycle boundaries (database open, migration) but **not at internal cast boundaries** (`node_identifier.js:22-28`). Every internal identifier originates from `makeNodeIdentifier()`, which produces valid strings by construction (fingerprint validated at boot, base-36 index from a monotonic counter). This is documented and consistent — the code trusts its own invariants.

### 1.2 Identifier Lookup (`database/identifier_lookup.js`, 562 lines)

The bijection between semantic keys and identifiers is managed by a new module with two data structures:

- **`IdentifierLookup`** (committed): In-memory mirror of the persisted `identifiers_keys_map`. Maintains `keyToId: Map<string, NodeIdentifier>` and `idToKey: Map<string, NodeKeyString>` + a cached `serialized: IdentifiersKeysMap` sorted array for efficient persistence.

- **`TransactionIdentifierLookup`** (overlay): Transaction-scoped overlay with its own `keyToId`/`idToKey` Maps, a read-only `base` reference to the committed lookup, and `ownedKeys: Set<string>` for cleanup tracking.

**Key design wins:**
- **No full clones**: The old code cloned the entire identifier map O(n) on every transaction. The overlay pattern makes transaction creation O(1).
- **Lazy serialization**: The `serialized` cache is rebuilt only when `serialized.length !== idToKey.size`; batch operations update it eagerly.
- **Disk-first ordering**: `serializeTransactionLookup()` runs *before* the disk flush (captures full state = base + overlay). `commitTransactionLookup()` runs *after* the flush succeeds (applies overlay to base in-memory). A failed flush leaves both disk and memory unchanged.

### 1.3 Graph State (`graph_state.js`, 515 lines — NEW, replaces `graph_storage.js`)

The old `graph_storage.js` (380 lines, deleted) mixed identifier management, batch operations, and lookup cloning into one monolithic facade. The replacement cleanly separates three concerns:

| Component | Role |
|---|---|
| **`BatchBuilder`** | Six handles (values, freshness, inputs, revdeps, counters, timestamps) with read-your-writes overlay |
| **`Transaction`** | A pair: `{ batch, identifierLookup }` — the lookup is an overlay, not a clone |
| **`GraphStorage`** | Factory-created object with `withTransaction`, `withBatch`, `ensureMaterialized`, etc. |

**Key improvement:** Revdep diffs are returned from the callback as `{ value, revdepDiffs? }` and applied during `darkroomActivity` at commit time, after the disk flush. This removes the O(n) clone of the old approach.

### 1.4 Fine-Grained Locking — The Cosmic Observatory (`lock.js`)

The old lock model had `MUTEX_KEY`, `GRAPH_ACTIVITY_KEY` (with `observe`/`pull` modes), and `PULL_NODE_KEY`. The new model uses a metaphor of a **cosmic observatory** with clearer semantics:

| Activity | Lock | What it protects |
|---|---|---|
| **daytimeActivity** | `DOME_CONDITION_DAYTIME` | Inspection: `getValue`, `getFreshness`, `listMaterializedNodes`, `invalidate` |
| **nighttimeActivity** | `DOME_CONDITION_NIGHTTIME` | Evaluation: `pull()` |
| **telescopeActivity** | Per-node mutex | One node's serialized pull (prevents duplicate identifier allocation) |
| **darkroomActivity** | Per-replica mutex | Commit finalization: disk flush + in-memory lookup update |
| **holidayActivity** | Exclusive (holiday gate + holiday mode) | Replica cutover, migration |

**Acquisition order (documented and enforced):** Pull: `dome nighttime` → `telescope(node)` → (optionally) `darkroom(replica)`. Holiday: `holiday gate` → `dome holiday condition`. The old `withMutex` (generic non-reentrant mutex) was removed entirely.

### 1.5 Sync Merge Decomposition

The old `sync_merge.js` (860 lines, monolithic) is now an orchestration layer (~558 lines) that delegates to five sub-modules:

| Module | Lines | Responsibility |
|---|---|---|
| `sync_merge.js` | 558 | Orchestration, `mergeHostIntoReplica()` workflow, error types |
| `sync_merge_plan.js` | 132 | Builds merge plan: timestamp comparison → initial decisions → merged inputs map → topological sort → taint propagation → final decisions |
| `sync_merge_transfer.js` | 81 | Data transfer: `copyReplicaGently()`, `buildTakeOps()` |
| `sync_merge_revdeps.js` | 77 | Reverse-dependency rebuilding from merged inputs map |
| `sync_merge_timestamps.js` | 22 | ISO-8601 timestamp comparison (undefined-as-oldest) |
| `sync_merge_identifier_lookup.js` | 74 | Host lookup parsing, conflict detection |

**Taint propagation** is the key algorithmic insight in `sync_merge_plan.js`: after initial keep/take decisions based on timestamps, the algorithm propagates "keepTainted" and "takeTainted" flags through the dependency graph in topological order. Nodes that receive both taints become `invalidate` — they must recompute from merged inputs.

### 1.6 Dual-Replica Architecture (`root_database.js`)

The root database now manages **two replicas** (`"x"` and `"y"`). At any time one is active (pointed to by `_meta/current_replica`). Synchronization and migration always:
1. Copy the active replica into the inactive one.
2. Perform the merge/migration on the inactive replica.
3. Atomically flip the pointer when changes occurred.

**`ActiveReplicaComputed`** is a new in-memory struct that mirrors the durable database state. It is reconstructable from disk and rebuilt on every replica switch. Ephemeral state (`_pendingAllocations`) lives outside this struct so it survives pointer changes.

### 1.7 Interface Changes (`graph_api.js`)

The old `withMutex`-protected invalidate+pair pattern was replaced by an `invalidateAndPull()` helper that:
1. Calls `ensureInitialized()` to get the graph.
2. Invalidates.
3. Calls `ensureInitialized()` again (gets re-initialized graph in case sync ran between).
4. Sets the value.
5. Pulls.

The three main operations (invalidate, set, pull) are deliberately **not treated as an atomic unit** — `synchronizeDatabase()` runs under `holidayActivity()` which cannot overlap with either phase. This is cleaner than the old approach of holding a global mutex across all three.

`ensureInitialized()` now returns `Promise<IncrementalGraph>` instead of `Promise<void>`, removing the need for a separate `_requireInitializedGraph()` call in most paths.

## 2. Migration Approach

### 2.1 Standalone Script (`scripts/migrate-snapshot-to-identifiers.js`, 459 lines)

The migration from old-format snapshots to the new identifier-based format is handled by a **standalone script**, not by runtime code. This is by design — the runtime contains zero old-format conversion logic.

**Flow:**
1. Idempotency check: skips if `identifiers_keys_map` already exists.
2. Walks old-format `values/`, `freshness/`, `inputs/`, etc., parsing path-based keys.
3. Loads/preserves fingerprint from `r/global/fingerprint`, or generates a new one.
4. Deterministic assignment: sorts entries by canonical ordering, assigns `index.toString(36)-fingerprint`.
5. Converts references in `inputs`/`revdeps` from old JSON key refs to identifiers (via `scripts/lib/convert_references.js`).
6. Cleans up old files and writes new metadata (`identifiers_keys_map`, `last_node_index`, `version`, `fingerprint`).

### 2.2 Migration Runner Changes (`migration_runner.js`, `migration_storage.js`)

Both modules were updated to use `NodeIdentifier` throughout instead of `NodeKeyString`. Key changes:
- `create()` now auto-generates identifiers via `makeNodeIdentifier()`.
- `resolveNodeKey()` is a new method for reverse lookup.
- `runMigration` returns `RootDatabase` instead of `void`.
- Uses `holidayActivity` instead of `withExclusiveMode`.

## 5. Release Safety

### 5.1 Release-Blocker Markers

The PR introduces **executable guardrails** for release safety:

- `scripts/check-release-blockers`: Scans tracked files for the marker `THIS-MARKER-BLOCKS-VOLODYSLAV-RELEASE-63461325`. Exits 1 if any found.
- `scripts/link-targeted`: Runs the blocker checker at installation time (unless bypassed via `VOLODYSLAV_ALLOW_RELEASE_BLOCKERS=1`).

**Current blocker:** `sync_merge.js` contains the marker with the note:
> "Release blocker: identifier-native sync does not yet repair the case where two hosts assign different NodeIdentifiers to the same semantic node key. The current merge policy rejects or avoids identifier conflicts but cannot merge diverged identifier assignments. This format must not be deployed until the conflict-resolution design is implemented. See issue #1410."

This is a known, documented, intentional blocker — the PR adds the infrastructure but acknowledges a remaining gap.

### 5.2 Safety Mechanisms

1. **Versioned storage**: All replicas carry a `version` in `global/version`. Version mismatch throws `HostVersionMismatchError`.
2. **Explicit migrations**: Old-format snapshots require explicit `migrate-snapshot-to-identifiers.js` — no silent repair.
3. **Identifier conflict detection**: `IdentifierLookupConflictError` provides clear failure mode.
4. **Disk-first ordering**: Volatile state never claims a fact the database doesn't yet contain.
5. **Staged synchronization**: Incoming state is validated before replica cutover.
6. **Atomicity**: All identifier and node-state updates are atomic within a single LevelDB batch.

## 6. Issues and Concerns

### 6.1 Release Blocker (Known, Documented)

The identifier conflict resolution gap (issue #1410) is a real concern. If two hosts independently assign different identifiers to the same semantic key, the merge hard-fails with `IdentifierLookupConflictError` and the message "Volodyslav will not resolve this automatically." This is correct but incomplete — a production deployment needs a resolution path. The blocker marker correctly prevents deployment until this is addressed.

### 6.2 Complexity Growth

The PR adds approximately **10,000 net new lines** of code. While much of this replaces old code (2,678 deletions), the total active codebase grows significantly. Key complexity drivers:
- **Dual-replica management**: Two complete sets of sublevels per hostname.
- **Five sync_merge sub-modules**: More files, more interfaces, more test surface.
- **Locking hierarchy**: Five activity types with documented acquisition orders that must be maintained by hand.
- **Transaction overlay pattern**: Correctness depends on callers never holding `base` references across `await`.

### 6.3 Async-Boundary Safety

The code relies on correctness of the locking discipline to prevent stale references. The `root_database.js` documentation includes a "stale-reference warning" explaining that callers must not capture sub-properties of `ActiveReplicaComputed` across `await`. While the lock hierarchy prevents concurrent replica switches during nightime/daytime activity, this is an **invariant that must be maintained by all callers** and is not enforced by the type system. This is a common pattern in the codebase (trust the runtime invariants) but worth noting.

### 6.4 Identifier Lookup Cache Consistency

The `serialized` cache in `IdentifierLookup` uses a **length-based staleness check** (`serialized.length !== idToKey.size`). This is correct as long as no mutation ever changes the size without touching the cache. The `commitTransactionLookup` and `mergeIdentifierLookups` functions update the cache explicitly. Individual mutations (`setIdentifierMapping`, `deleteIdentifierMappingForNodeKey`) only mutate Maps. The length check catches staleness after individual mutations but requires that the cache is always exactly in sync after batch operations. This seems correct but is a fragile invariant.

### 6.5 Migration Script Error Handling

The migration script (`migrate-snapshot-to-identifiers.js`) makes specific assumptions about the input format (tilde encoding for non-string arguments, specific directory structure). If a snapshot deviates from these assumptions in unexpected ways, the error messages may be opaque. Some robustness improvements (e.g., better error messages for unexpected formats) would be helpful but not blocking.

## 7. Summary

This is a well-structured, well-documented refactor that significantly improves the architecture:

| Aspect | Before | After |
|---|---|---|
| Identifiers | Opaque random strings | Deterministic `<index>-<fingerprint>` |
| Identifier management | Cloned on every transaction | Overlay pattern, O(1) transaction creation |
| Locking | `MUTEX_KEY` + observe/pull modes | Cosmic observatory: 5 activity types with documented hierarchy |
| Graph storage | Monolithic `graph_storage.js` | Clean separation: `graph_state.js` with `BatchBuilder`/`Transaction`/`GraphStorage` |
| Sync merge | 860-line monolith | 6 focused modules |
| Replicas | Single replica | Dual-replica with atomic cutover |
| Documentation | Minimal | Comprehensive specs: lifecycle, locking, consistency, fingerprint, release safety |
| Release safety | None | Versioned storage, release-blocker markers, executable guardrails |

**Recommendation:** The code quality is high, the documentation is thorough, the test coverage is substantial, and the release blocker is correctly preventing premature deployment of the remaining gap. The PR should be merged after:
1. The known release blocker (identifier conflict resolution, issue #1410) is addressed, or the decision to proceed with the blocker is explicitly made.
2. The `serialized` cache staleness check is verified to be correct under all mutation patterns (this should be a quick audit, not a redesign).
