
### 6.3 Async-Boundary Safety

The code relies on correctness of the locking discipline to prevent stale references. The `root_database.js` documentation includes a "stale-reference warning" explaining that callers must not capture sub-properties of `ActiveReplicaComputed` across `await`. While the lock hierarchy prevents concurrent replica switches during nightime/daytime activity, this is an **invariant that must be maintained by all callers** and is not enforced by the type system. This is a common pattern in the codebase (trust the runtime invariants) but worth noting.

### 6.4 Identifier Lookup Cache Consistency

The `serialized` cache in `IdentifierLookup` uses a **length-based staleness check** (`serialized.length !== idToKey.size`). This is correct as long as no mutation ever changes the size without touching the cache. The `commitTransactionLookup` and `mergeIdentifierLookups` functions update the cache explicitly. Individual mutations (`setIdentifierMapping`, `deleteIdentifierMappingForNodeKey`) only mutate Maps. The length check catches staleness after individual mutations but requires that the cache is always exactly in sync after batch operations. This seems correct but is a fragile invariant.

### 6.5 Migration Script Error Handling

The migration script (`migrate-snapshot-to-identifiers.js`) makes specific assumptions about the input format (tilde encoding for non-string arguments, specific directory structure). If a snapshot deviates from these assumptions in unexpected ways, the error messages may be opaque. Some robustness improvements (e.g., better error messages for unexpected formats) would be helpful but not blocking.
