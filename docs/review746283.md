## Summary

This PR changes incremental-graph persistence so that materialized graph nodes are stored under stable generated `NodeIdentifier`s rather than semantic node-key paths. It introduces a persisted allocation namespace (fingerprint) and a monotonic allocation watermark (`last_node_index`), moves graph-state sublevels to identifier-addressed storage, adds a bijection lookup table (`identifiers_keys_map`), and updates every storage path â€” render, scan, sync, reset/import, migration â€” to preserve the new model.

The PR body explicitly states this is **not intended for production release** until proper conflicting-identifier sync handling is implemented (issue #1410). That frames the review appropriately: issues that are acceptable for a foundational draft may become blockers before production release.

## What works well

### Lifecycle model (`database-lifecycle.md`)
The new lifecycle specification is the strongest addition. It defines six states (Absent, Current, Migratable, Incompatible, Corrupted/Unsupported), six supported transitions, and a clear corruption model: "state not produced by a supported Volodyslav transition." Cross-host snapshot cloning is explicitly excluded from the supported model (Â§10). This defuses the fingerprint-collision concern without requiring clustering-level coordination.

### Fingerprint design
The fingerprint lives in replica-global metadata (`r/global/fingerprint`) rather than `_meta`, which means restore/import/reset paths scan only `rendered/r/` and do not need a special `_meta` scan path. Three generation paths are defined: fresh boot creates new; first-boot restore from snapshot imports (host recovering its own state); existing-DB reset preserves local. Runtime validation via `requireValidFingerprint()` at lifecycle boundaries ensures format correctness without adding validation overhead to hot internal paths.

### Transaction model
The volatile-consistency spec is comprehensive and self-consistent. The overlay-based `TransactionIdentifierLookup` avoids full-map clones on every transaction. The commit sequence is: serialize â†’ disk batch â†’ only then publish to volatile state. The disk-before-memory invariant is maintained throughout. The `serialized` cache on `IdentifierLookup` avoids re-iterating and re-sorting on every commit.

### Public API hygiene
`IncrementalGraphClass` now exposes only user-targeted methods (`pull`, `invalidate`, `getFreshness`, `getValue`, `getSchemas`, `getSchemaByHead`, `listMaterializedNodes`, `getDbVersion`, `getCreationTime`, `getModificationTime`). Internal machinery (`withTransaction`, `resolveConcreteNode`, `_pullDuringPull`, `lookupNodeIdentifier`, `unsafePull`, `pullWithStatus`, `unsafeInvalidate`, `propagateOutdated`) has been removed from the class and lives in the calling modules (`pull.js`, `invalidate.js`, `recompute.js`). Transaction callers go through `storage.withTransaction()` directly, and the required locking discipline is documented in the `withTransaction` JSDoc.

### Crash-atomic initialization
`makeRootDatabase()` now writes all active-replica metadata (`identifiers_keys_map`, `last_node_index`, `fingerprint`) before writing `_meta/current_replica` as the final commit step. If a crash occurs before the pointer write, the next open sees no active replica pointer and re-initialises cleanly.

### Sync merge
The merge algorithm is conservative: it checks identifier conflict (`assertNoIdentifierLookupConflicts`) before the expensive copy phase, fails fast on version mismatch, preserves the local fingerprint and `last_node_index` watermark, and only commits when graph records actually change (not on metadata-only host differences).

### Test coverage
New tests cover fingerprint design (`fingerprint_design.test.js`, 28 tests), identifier lookup correctness (`identifier_lookup.test.js`, `identifiers_keys_map_correctness.test.js`), identifier resolution (`identifier_resolver.test.js`), materialized-node invariant (`materialized_node_invariant.test.js`), volatile consistency (`incremental_graph_volatile_consistency.test.js`), migration script compatibility (`migration_script_compatibility.test.js`), and expanded sync merge (`sync_merge.test.js`).

## Design concerns

### 1. `IdentifierLookup.serialized` cache coherency â€” fragile implicit contract

Three operations update the `serialized` cache: `makeIdentifierLookup`, `commitTransactionLookup`, and `mergeIdentifierLookups`. Two operations do **not**: `setIdentifierMapping` and `deleteIdentifierMappingForNodeKey`. A caller using those independently would leave a stale cache. The current callers all go through the cache-aware entry points, but there is no guard or assertion if someone calls the wrong function.

Also, `cloneIdentifierLookup()` shares the `serialized` reference by identity. If the source lookup is mutated after cloning, the clone's serialized form becomes stale. This is currently safe because clones are short-lived in controlled paths, but it is a subtle footgun.

**Acceptable for draft.** Add an `_unused` validation flag or assertion on `serialized` before release.

### 2. Encapsulation leak: `graph_state.js` accesses `RootDatabaseClass` private method

`withTransaction` in `graph_state.js` calls `rootDatabase._releaseAllocations(txLookup.ownedKeys)`. The `_releaseAllocations` method is conceptually private to `RootDatabaseClass`. While the two modules are in the same submodule system, this breaks the strict layered architecture described in the volatile-consistency spec (where `graph_state.js` depends on `database/` but should not reach into implementation details).

**Acceptable for draft.** Either promote `_releaseAllocations` to a documented method or introduce a narrow public API (`releaseIdentifierReservations(ownedKeys)`) before production release.

### 5. `sync:false` crash-recovery window in unification adapters

The unification adapters use `sync:false` per-write and a single `_rawSync()` (empty batch with `sync:true`) at the end. This relies on LevelDB's WAL semantics. A crash between the last per-write and the sync call that also loses the WAL (e.g., OS crash without fsync) could leave partial state in the database. LevelDB's default WAL sync behavior mitigates this for typical process crashes, but the gap is theoretical.

**Not a blocker.** Well-documented in the code.

## Code-level observations

### `serializeIdentifierLookup` now returns cached `serialized`
The cache was always maintained by `makeIdentifierLookup`, `commitTransactionLookup`, and `mergeIdentifierLookups`, but `serializeIdentifierLookup` was rebuilding from Maps every time. Now it returns `lookup.serialized` directly. The comment documents that all mutation paths keep the cache up to date.

### Sync merge no longer max-merges `last_node_index`
The original design (`mergedLastNodeIndex = max(target, host)`) was removed. The merge preserves the local watermark unchanged. Host `last_node_index` differences alone do not cause a merge commit or replica switch. The spec now documents this correctly.

### Fingerprint spec was updated for cross-host cloning
Earlier versions implied each machine always has its own fingerprint. The spec now explicitly says cross-host snapshot cloning is unsupported and documents that sync's identifier conflict check catches it cleanly if attempted.

### `pull.js` imports `internalMaybeRecalculate` from `recompute.js`
This creates a dependency from `pull.js` â†’ `recompute.js`. The reverse call (dependency pulls during recomputation) is broken by passing a `pullDependency` callback as an explicit parameter to `internalMaybeRecalculate`, avoiding the circular module dependency.

## Verdict

**Merge as draft.** The design is sound, the specifications are thorough and self-consistent, and the implementation matches them. The PR's draft status and stated deferral of issue #1410 mean that the remaining concerns (cache coherency contracts, encapsulation leakage, retry strategy) are acceptable for a foundational change of this size.

Before promoting to a production-ready merge request, address:
1. Cache coherency contract hardening for `IdentifierLookup.serialized`
2. The `_releaseAllocations` encapsulation boundary
3. Exponential backoff in the retry loop
4. Whether `isValidNodeIdentifier` should be removed or gated
