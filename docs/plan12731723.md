# Plan: Rewrite Incremental Graph for Volatile Consistency

## Reference

Spec: `docs/volatile-consistency-of-incremental-graph.md`

---

## Problem Statement

The incremental graph implementation allows concurrent pull/invalidate flows that each create their
own independent `IdentifierResolver` snapshot outside the `withComputedStateMutex` critical section.
When committing, the implementation merges the resolver's pending allocations back onto the current
active lookup (`applyPendingTo(getActiveLookup(...))`). This merge is the root of the divergence
problem described in the spec.

### Root Cause

1. `makeIdentifierResolver(rootDatabase)` is called **before** `withComputedStateMutex` is
   acquired. The resolver lazily loads the active identifier lookup at first use (also before or
   during the critical section, depending on timing).

2. Inside `withIdentifierBatch`, after acquiring the mutex, the commit step re-reads the active
   lookup (which may have advanced since the resolver loaded its snapshot) and then merges the
   resolver's new allocations on top. This is the "merge recovery logic" the spec wants to remove.

3. The merge can cause races: two concurrent resolvers that both loaded the pre-mutex state and both
   allocated new identifiers will merge independently, potentially with divergent snapshots.

### Spec Target

> Treat one `_computed` state (active replica) as the serialization scope for computed-state edits.
> Run each top-level computed edit under that lock.
> Persist resolver lookup as a single snapshot from the active operation, instead of merging
> independent concurrent snapshots.
> Nested dependency pulls reuse the same resolver edit scope and do not reacquire the
> `_computed` lock.

---

## Design

### Core Invariant

**The `IdentifierResolver` must always be created inside `withComputedStateMutex`.**

This guarantees:
- The resolver loads the active lookup after `ensureActiveIdentifierLookupLoaded()`, seeing a
  fully consistent, up-to-date snapshot.
- Any new identifier allocations are made against this correct snapshot.
- At commit time, `identifierResolver.lookup` is the authoritative, complete, correct state:
  `base_lookup + new_allocations`. No merge is needed.
- Concurrent top-level edits are fully serialized end-to-end: lock acquisition → resolver creation
  → computation → disk flush → active-lookup update → lock release.

### Re-entrancy for Nested Pulls

Nested pulls (triggered from inside a computor) reuse the outer resolver and batch via the
existing `pushActivePullContext`/`getActivePullContext`/`popActivePullContext` async-context
mechanism. They do NOT call `withIdentifierBatch` again and do NOT acquire the mutex again. This
prevents deadlock and ensures all allocations in a single top-level edit are in one resolver
snapshot.

### API Changes

#### `GraphStorage.withIdentifierBatch`

Old signature:
```
withIdentifierBatch(identifierResolver, fn: (batch) => Promise<T>) => Promise<T>
```

New signature:
```
withIdentifierBatch(fn: (batch, identifierResolver) => Promise<T>) => Promise<T>
```

The function now:
1. Acquires `withComputedStateMutex` for the current replica.
2. Calls `ensureActiveIdentifierLookupLoaded()` if available.
3. Creates `identifierResolver = makeIdentifierResolver(rootDatabase)` inside the critical section.
4. Calls `fn(batch, identifierResolver)`.
5. If `identifierResolver.hasPendingAllocations`, persists `identifierResolver.lookup` (a clone)
   directly — no merge. Then flushes to disk. Then updates the volatile active lookup.
6. Releases the mutex.

#### `IdentifierResolver`

Remove `applyPendingTo` (the merge helper that enabled the old pattern). Also remove the internal
`pendingIdentifierMappings` map (was only used by `applyPendingTo`).

The `lookup` getter still returns the mutable in-memory lookup snapshot (base + all allocations
made so far). `hasPendingAllocations` remains.

## Step-by-Step Changes

### Step 1 — `identifier_resolver.js`

- Remove `applyPendingTo` from the returned resolver object.
- Remove `pendingIdentifierMappings` from internal state (was only used by `applyPendingTo`).
- Update the `IdentifierResolver` JSDoc typedef to omit `applyPendingTo`.

No other behaviour changes. `hasPendingAllocations`, `lookup`, `lookupNodeIdentifier`,
`getOrAllocateNodeIdentifier`, `requireNodeKey` are all unchanged.

### Step 2 — `graph_storage.js`

- Import `makeIdentifierResolver` from `./identifier_resolver`.
- Remove `getActiveLookup` from the imports (no longer used here).
- Change `withIdentifierBatch`:
  - Drop the `identifierResolver` parameter.
  - Create the resolver with `makeIdentifierResolver(rootDatabase)` after the mutex is held and
    `ensureActiveIdentifierLookupLoaded()` has been called.
  - Pass `(batch, identifierResolver)` to `fn`.
  - On commit: if `identifierResolver.hasPendingAllocations`, clone `identifierResolver.lookup`
    and use that as `lookupToCommit`. No call to `getActiveLookup`. No `applyPendingTo`.
- Update the `GraphStorage` JSDoc typedef for `withIdentifierBatch` to reflect the new signature.

### Step 3 — `class.js`

- Remove the `makeIdentifierResolver` import and the `makeIdentifierResolver()` method on the
  class (no longer needed externally).
- Update the `withIdentifierBatch(procedure)` method:
  - Drop the `identifierResolver` parameter.
  - Just forward to `this.storage.withIdentifierBatch(procedure)`.
- Add a `lookupNodeKey(nodeIdentifier)` method:
  - Returns `this.rootDatabase.nodeIdToKey(nodeIdentifier)` (read-only, outside mutex).
  - Used by `internalPullByNodeIdentifierWithStatusDuringPull` for top-level identifier→key
    resolution without needing a resolver.

### Step 4 — `pull.js`

- Update `IncrementalGraphPullAccess` typedef:
  - Remove `makeIdentifierResolver` property.
  - Add `lookupNodeKey: (nodeIdentifier: NodeIdentifier) => NodeKeyString | undefined` property.
  - Update `withIdentifierBatch` signature to reflect new API.

- `internalPullWithStatus`:
  - Remove the call to `incrementalGraph.makeIdentifierResolver()`.
  - Pass no resolver to `internalPullByNodeKeyWithStatusDuringPull` (null / omitted).

- `internalPullByNodeKeyWithStatusDuringPull`:
  - Change default for `identifierResolver` from `makeIdentifierResolver()` to `null`.

- `runPullForSemanticNodeKey`:
  - Accept `identifierResolver` as nullable (null for top-level, non-null for explicit nested).
  - For the top-level path (no outer batch): call
    `incrementalGraph.withIdentifierBatch(async (batch, identifierResolver) => runWithContext(batch, identifierResolver))`.
  - Refactor the inner `run` closure into a named `runWithContext(batch, identifierResolver)`
    that is called in both the nested and top-level paths.

- `internalPullByNodeIdentifierWithStatusDuringPull`:
  - Remove external `identifierResolver` parameter.
  - If inside an active pull context: use `activePullContext.identifierResolver.requireNodeKey()`
    to resolve the identifier, then call `runPullForSemanticNodeKey` with that context's
    resolver and batch.
  - Otherwise (top-level): use `incrementalGraph.lookupNodeKey(nodeIdentifier)` to read from the
    active lookup snapshot (lock-free read), then call `runPullForSemanticNodeKey` with null
    resolver and null batch (causing `withIdentifierBatch` to create the resolver inside mutex).

### Step 5 — `invalidate.js`

- Update `IncrementalGraphInvalidateAccess` typedef:
  - Remove `makeIdentifierResolver` property.
  - Update `withIdentifierBatch` signature.

- `internalUnsafeInvalidate`:
  - Remove the call to `incrementalGraph.makeIdentifierResolver()`.
  - Pass the `withIdentifierBatch` callback as `async (batch, identifierResolver) => { ... }`.
  - Use the `identifierResolver` received inside the callback for all resolver operations.

### Step 6 — Tests

- `backend/tests/identifier_resolver.test.js`:
  - Remove the `IdentifierResolver applyPendingTo` describe block entirely (the feature is gone).
  - Add a test verifying that `resolver.lookup` contains all allocations after
    `getOrAllocateNodeIdentifier` calls (tests the invariant `withIdentifierBatch` now relies on).

- `backend/tests/incremental_graph_concurrency.test.js`:
  - Update the two tests that monkey-patch `graph.storage.withIdentifierBatch`:
    - Change mock signature from `async (resolver, run) => ...` to `async (run) => ...`.
    - Update the call to `originalWithIdentifierBatch` accordingly.

## Invariants Preserved

1. **Single snapshot per top-level edit**: the resolver is created once inside the mutex; all
   identifier work in that edit (including nested pulls) flows through the same snapshot.
2. **No merge**: `identifierResolver.lookup` is persisted directly after being mutated in-place by
   allocations.
3. **Disk-first ordering**: the disk flush happens before `setActiveLookup` (unchanged).
4. **Lock-free reads**: `getValue`, `getFreshness`, `listMaterializedNodes`, and
   `internalPullByNodeIdentifierWithStatusDuringPull` (identifier→key step) are still lock-free.
5. **Nested pull re-entrancy**: existing `pushActivePullContext`/`getActivePullContext` mechanism
   is unchanged; nested pulls still share the outer resolver and batch without reacquiring the
   mutex.
6. **Rollback on flush failure**: if the batch flush throws, `setActiveLookup` is never called;
   the volatile lookup remains at the pre-edit state (unchanged invariant).
