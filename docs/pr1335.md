# PR #1335 — Switch IncrementalGraph persistence and migration to node identifiers

## Links
- PR: https://github.com/ottojung/volodyslav/pull/1335
- Review thread called out in follow-up: https://github.com/ottojung/volodyslav/pull/1335#pullrequestreview-4324712985

## High-level intent
This PR migrates IncrementalGraph persistence from **semantic node keys** to **opaque node identifiers**. The conceptual objective is to decouple persisted graph state from semantic naming details so storage identity remains stable and compact while semantic meaning is tracked through a dedicated lookup map.

At a systems level, the change introduces a persisted bijection map at:

- `/<replica>/global/identifiers_keys_map`

that links:

- `NodeIdentifier` (opaque persisted key)
- `NodeKeyString` (semantic key serialization)

## Architectural changes
1. **Identifier lookup as first-class persisted metadata**
   - Added lookup utilities (`makeIdentifierLookup`, `mergeIdentifierLookups`, allocation, serialization, conflict checks).
   - Added explicit error classes for lookup corruption/collisions.

2. **Database layer alignment**
   - Root schema and typed database paths now recognize `identifiers_keys_map` as global metadata.
   - Encoding/decoding and db/fs unification paths were updated so identifier-keyed records and lookup metadata move together.

3. **Runtime graph operations now identifier-native**
   - Graph persistence tables (`values`, `freshness`, `inputs`, `revdeps`, `counters`, `timestamps`) are keyed by `NodeIdentifier`.
   - Resolver flow bridges semantic keys to identifiers when needed.

4. **Sync/merge behavior updated**
   - Host/target merges include lookup-map merging into target global metadata.
   - Revdeps unification is computed from merged inputs state.

5. **Migration path extended**
   - Migration runner supports both legacy states (without map) and identifier-native states (with map).
   - Key-planning logic remaps keys for callback compatibility and output persistence.

## Low-level implementation choices
- **Strict bijection invariant:** lookup map enforces one-to-one mapping both directions; conflicting inserts raise `IdentifierLookupError`.
- **Deterministic identifier derivation for migration:** hash-based deterministic fallback for legacy key materialization.
- **Sorted serialization:** persisted lookup entries are sorted by identifier for deterministic storage.
- **Error typing strategy:** specific classes and type guards are used instead of generic errors.
- **Compatibility boundary:** migration wrapper keeps callback API shape stable even though persisted keys are now identifiers.

## Behavior impact summary
- Persisted data is now anchored to opaque identifiers.
- Semantic key understanding moves to lookup metadata.
- Merge and migration correctness now depend on robust lookup reconciliation and persistence.

## Known feedback themes from review cycle
The review stream surfaced two correctness gaps (expanded in `docs/pr1335-review1.md`):
- independent identifier allocation collisions across replicas during sync merge,
- missing map persistence for newly created nodes during legacy migrations.
