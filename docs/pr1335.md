# PR #1335 Study: Switch IncrementalGraph persistence and migration to node identifiers

## High-level intent
PR #1335 migrates IncrementalGraph persistence from semantic node keys (`head + args`) to stable node identifiers. Conceptually, this decouples runtime/persistent graph state from semantic key encoding and centralizes key↔identifier mapping in the identifier lookup layer.

At a system level, this has three goals:

1. Reduce repeated semantic-key parsing/serialization in storage paths.
2. Make migration logic explicit for legacy snapshots that still store key-addressed records.
3. Move graph storage APIs toward identifier-native operations.

## Conceptual design choices observed in current branch

### 1) Identifier-centric storage schema
The storage-facing code in incremental graph state now primarily treats node identity as `NodeIdentifier`, with `NodeKeyString` used at API boundaries and lookup/migration boundaries.

### 2) Transaction overlay for identifier allocation
`graph_state.withTransaction` uses an overlay lookup (`makeTransactionIdentifierLookup`) and commits in disk-first order:
- collect batch ops and new allocations,
- flush to persistent storage,
- then publish lookup overlay into volatile state.

This is a deliberate consistency choice: volatile lookup must not advance ahead of persisted state.

### 3) Pull/invalidate/read phase partitioning
The lock layer separates graph activity into pull vs observe modes (`withPullMode` / `withObserveMode`) and keeps an exclusive mode for migrations/open-like operations (`withExclusiveMode`).

## Low-level implementation details

- `pull.js` executes top-level pulls under pull mode and routes nested pulls through explicit transaction passing.
- `invalidate.js` and `inspection.js` run under observe mode and operate over identifiers in storage.
- `graph_state.js` transaction commit persists identifier deltas under `IDENTIFIERS_KEY` when allocations occurred.
- `lock.js` includes global graph activity mode-locking plus a computed-state mutex used by transaction execution.

## Risks and trade-offs already visible

1. **Concurrency vs allocation safety**: current transaction locking serializes all transactional callers to avoid identifier allocation races.
2. **Spec drift risk**: global transaction serialization may conflict with the intended per-node pull concurrency semantics in locking design docs.
3. **Migration correctness burden**: legacy key-addressed records must be transformed consistently across all graph indexes.

## Summary
PR #1335 is structurally moving the graph toward identifier-first persistence with stronger migration discipline and explicit lock phases, but it also introduces (or retains) conservative serialization in transaction execution that needs explicit validation against the locking spec.
