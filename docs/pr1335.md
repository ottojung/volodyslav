# PR #1335 — Incremental Graph Identifier-Native Persistence, Replica Switching, and Volatile/Persistent Consistency

## High-level summary
PR #1335 is a large refactor of the incremental-graph persistence model. The core change is this:

- **Before**: many persisted structures were keyed by transient/internal node keys.
- **After**: persistence is normalized around **stable node identifiers** and a dedicated lookup map.

Conceptually, the PR is trying to make the database:
1. More stable across process lifetimes.
2. More robust during migrations and pull/sync flows.
3. Better aligned with the runtime graph semantics (especially under batching/transactions).

## Main conceptual shifts

### 1) Identifier-native storage
The storage layers for values, freshness, inputs, revdeps, counters, and timestamps are moved toward identifier-centric indexing.

Why this matters:
- Identifiers are designed to be durable/stable.
- It reduces ambiguity when internal key allocation changes.
- It makes migrations/reconciliation logic more explicit.

### 2) Global identifier lookup as a first-class record
A global `identifiers_keys_map`/lookup mechanism becomes a key piece of the schema.

Why this matters:
- Node-key ↔ identifier translation is now explicit persistence, not implied runtime state.
- Pull/migration/sync steps can reason over a single source of truth.

### 3) Replica-cutover and synchronization hardening
The PR includes multiple rounds of updates around active-replica pointer handling and atomicity guarantees.

Why this matters:
- Switching active replicas is high risk for data integrity.
- The PR attempts to improve all-or-nothing behavior for replica pointer updates.

### 4) Volatile vs persistent consistency model
There is a significant evolution in how in-memory/volatile graph state is coordinated with persisted state, especially around transaction/batch boundaries.

Why this matters:
- Incremental graph correctness depends on deterministic updates between cache-like volatile state and durable storage.

## Low-level decisions that stand out

1. **Lookup persistence placement evolved over the PR**
   - Commits indicate a redesign to keep identifier-map writes within the same batch/transaction as related node writes.
   - This improves atomicity but also increases coupling between resolver/storage layers.

2. **Extensive migration plumbing**
   - New migration runner/storage logic and fixture updates were introduced.
   - Multiple follow-up fixes suggest migration edge cases were discovered late.

3. **Sync/pull context tracking complexity increased**
   - Later commits optimize nested async pull-context handling and ordering assumptions.
   - This likely fixed race/ordering bugs, but it also introduces more subtle concurrency invariants.

4. **A lot of patch churn and temporary-doc commits**
   - The PR series contains repeated “remove tmp docs” and “potential fix” commits.
   - The final shape may be correct, but reviewability and rationale traceability are degraded.

## Risk map

### Correctness risks
- Identifier-map drift from node stores if any batch boundaries are inconsistent.
- Replica pointer cutover bugs leading to split-brain reads/writes.
- Out-of-order async context completion in pull/sync flows.

### Maintainability risks
- Very large scope across storage core + migration + runtime behavior.
- Implicit invariants spread across modules (resolver, storage, migration runner, sync merge).
- Hard for new contributors to infer the expected transactional contract.

## Suggested “mental model” for reading this PR
Read it in this order:
1. **Data model contracts** (identifier types, lookup structures, persisted sublevels).
2. **Write atomicity model** (what must commit together).
3. **Migration guarantees** (what legacy states are expected and how converted).
4. **Replica cutover contract** (when active pointer changes and what must already be true).
5. **Concurrency assumptions** (nested pull/sync context lifecycle).

This order avoids getting lost in commit-level churn and keeps the focus on invariants.
