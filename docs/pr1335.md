# PR #1335: Switch IncrementalGraph persistence and migration to node identifiers

## Context and high-level objective
PR #1335 (`Switch IncrementalGraph persistence and migration to node identifiers`) moved IncrementalGraph storage and migration behavior away from semantic node-key persistence and toward opaque, stable node identifiers. The core intent was to decouple persistence internals from semantic key structure (`head + args`) and reduce repeated key↔identifier translation in hot paths.

At a conceptual level, this PR treats node identifiers as the canonical storage identity, while semantic node keys become a lookup concern at the boundaries where callers request nodes by semantics.

## Architectural direction

The PR's direction can be summarized as:

1. **Identifier-native storage model**
   - Persist graph state indexed by node identifier, not semantic key strings.
   - Keep semantic meaning in a separate identifier lookup bijection.

2. **Migration compatibility for legacy data**
   - Migrate historical snapshots/records that used older keying assumptions into the new identifier-based layout.
   - Preserve determinism for migration identifiers when possible.

3. **Boundary-focused translations**
   - Perform semantic-key → identifier resolution at API/operation boundaries.
   - Keep core storage operations as identifier-only operations.

## Low-level choices that matter

From the implementation footprint and follow-up commits in this branch history, notable low-level choices include:

- Introduction/usage tightening around identifier lookup helpers in `backend/src/generators/incremental_graph/database/identifier_lookup.js`.
- Refactoring call paths in sync and migration code so identifier lookups are explicit and centralized, including `sync_merge.js` and migration runner flows.
- Preservation and restoration of typing/comments after review feedback, indicating a deliberate attempt to keep JSDoc precision and maintainability.
- Multiple stabilization passes after #1335 landed, showing that the change was architecturally correct but had edge-case correctness gaps in reconciliation and cutover flows.

## Why this PR was important

The PR was not just a representation change; it was an **invariant change**:

- Before: storage and semantics were more tightly coupled.
- After: storage identity is opaque and semantic mapping is managed separately.

That enables cleaner internals, fewer accidental semantic assumptions in storage code, and better long-term flexibility for graph evolution.
