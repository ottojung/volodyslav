# PR #1335: Switch IncrementalGraph persistence and migration to node identifiers

## High-level goal
PR #1335 changes IncrementalGraph persistence from semantic node keys (`head + args`) to nominal `NodeIdentifier`s as primary storage keys. The architectural intent is to make graph runtime operations identifier-native after initial resolution, reduce key↔id translations, and make persistence/migration robust across semantic-key evolution.

## Conceptual model shift
- **Before**: core graph storage mixed semantic keys and operational identifiers.
- **After**: low-level storage (`graph_storage.js`) is identifier-only; semantic key handling is pushed to higher-level resolution boundaries.
- **Migration**: legacy snapshots keyed by semantic keys are transformed to identifier-keyed records and index consistency is restored.

## Low-level technical choices
- Introduces/leans on typed nominal representations for `NodeIdentifier` and conversion helpers.
- Uses identifier-keyed sublevels for values, freshness, inputs, reverse deps, counters, and timestamps.
- Keeps batch semantics with read-your-writes overlays to guarantee deterministic behavior within write transactions.
- Preserves and/or restores rich JSDoc typedef documentation so storage contracts remain explicit.

## Review-driven constraints from the PR discussion
Key owner feedback on May 16, 2026 emphasized:
1. Keep comments/typedefs comprehensive and useful.
2. Ensure `graph_storage.js` has no semantic-key (`NodeKey`) knowledge.
3. Do not delete tests; fix implementation first and rewrite tests equivalently if necessary.
4. Resolve static-analysis and test failures cleanly.
