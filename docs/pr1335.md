# PR #1335 Deep Dive

## High-level intent
PR #1335 migrated incremental-graph database key handling from a mixed/hybrid representation toward **identifier-native persistence**. The goal was to make key semantics explicit at the storage boundary, reduce ambiguous conversions, and ensure DB↔filesystem snapshot transforms remain reversible.

Conceptually, the PR did three things:

1. Reframed key handling around stable node identifiers rather than ad-hoc key blobs.
2. Split database rendering/scanning logic into clearer modules so encoding/decoding responsibilities are local and testable.
3. Tightened migration behavior so legacy snapshots still load while newly rendered snapshots preserve the newer canonical layout.

## Main conceptual changes

### 1) Key encoding as an explicit contract
The key encoder/decoder became a first-class interface for filesystem snapshots:
- raw DB key (`!x!!values!{"head":"event","args":["id"]}`)
- relative path (`x/values/event/id`)
- inverse mapping back to raw DB key

This made path-level compatibility and reversibility a formal requirement, especially for:
- separator-sensitive chars (`!`, `/`, `%`)
- dot segments (`.`/`..`)
- mixed arg types (string, number, boolean, object, array)

### 2) NodeKey-aware vs plain-key sublevels
A crucial low-level design choice: sublevels are split into two groups.
- **Plain-key sublevels** (e.g. `_meta`, `global`) store a single encoded string segment.
- **NodeKey sublevels** (`values`, `freshness`, etc.) deserialize NodeKey JSON and map `head + args` to path segments.

This avoids accidental conflation between metadata keys and graph-node keys.

### 3) Migration and typing cleanup
Several follow-up commits in the PR stack tightened typing boundaries around identifiers and conversion APIs. The direction was to keep conversions at boundaries, not spread across business logic.

## Low-level choices worth calling out

- Percent-encoding order preserves bijection (`%` before `/`/`!`).
- Sentinel encoding for unsafe path segments (`%2E`, `%2E%2E`, `%00`) prevents traversal ambiguity.
- Non-string NodeKey args use a `~` prefix with JSON payload; string args beginning with `~` are escaped as `~~...`.
- Invalid NodeKey JSON in NodeKey sublevels is treated as an error (not silently accepted), protecting snapshot correctness.

## Why this matters
This PR is fundamentally about making snapshot rendering/scanning deterministic and safe to refactor. If this boundary is weak, migrations and tooling can silently corrupt graph state.
