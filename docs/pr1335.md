# PR #1335 — Switch IncrementalGraph persistence and migration to node identifiers

## High-level overview

PR #1335 replaces the on-disk storage key for every IncrementalGraph node from a human-readable,
semantically-derived key (a `NodeKeyString` encoding the node's head name + argument list) to an
opaque, randomly-generated **node identifier** (a `NodeIdentifier`).

### Motivation

The previous scheme stored graph data using keys that encoded computation semantics. This created
several problems:

- **Merge fragility.** When branches from different hosts are merged, their data must be reconciled.
  Semantic keys are verbose and differ by host when the same logical node has slightly different
  key representations, causing spurious conflicts.
- **Migration brittleness.** Changing a node's definition or argument encoding required touching
  every persisted key for that node, making schema migrations invasive.
- **Storage inefficiency.** Semantic keys are long strings; opaque identifiers are a fixed 9 bytes.

By switching to opaque identifiers the storage layer becomes **content-addressed**: identifiers are
stable and meaningless strings. The mapping between semantic keys and identifiers is maintained in a
separate bijection (the *identifier lookup*) that lives both in-memory and on disk.

### Key concepts introduced

| Concept | Description |
|---------|-------------|
| `NodeIdentifier` | A 9-character lowercase-ASCII string used as the on-disk key for every node. |
| `IdentifierLookup` | An in-memory bijection `NodeKeyString ↔ NodeIdentifier`, loaded from and persisted to the `identifiers_keys_map` entry in the replica's global sublevel. |
| `IdentifierResolver` | A per-operation object that lazily snapshots the active lookup, caches translations, and tracks any newly allocated identifiers. |
| `withIdentifierBatch` | The atomic commit primitive: runs graph-level operations and then, under the `withComputedStateMutex`, merges any new identifier allocations and flushes everything to LevelDB in one batch. |

### Principal structural changes

1. **`graph_storage.js` is now identifier-native.** Every read and write uses `NodeIdentifier` as the
   key type. The module has no knowledge of `NodeKeyString` or semantic keys.
2. **`identifier_resolver.js` owns all semantic-to-identifier translation.** Translation happens
   exactly once per key per top-level IncrementalGraph operation; subsequent accesses within the
   same operation use a resolver-local cache.
3. **`RootDatabaseClass._computed` gains an `identifierLookup` field.** This is the authoritative
   in-memory bijection for the active replica.
4. **Migration runner converts legacy entries.** During the first startup after the PR is applied,
   existing key-addressed data is migrated to identifier-addressed data by deriving a deterministic
   identifier for each node (SHA-256 hash of the key).

## Low-level design details

### `NodeIdentifier` (database/node_identifier.js)

- Represented as a 9-character string of lowercase ASCII letters (`a`–`z`).
- Validated at construction by `nodeIdentifierFromString`, which rejects any value that does not
  match the 9-lowercase-letter pattern.
- Generated either randomly (via `NonDeterministicSeed`) or deterministically (via
  `deterministicNodeIdentifierFromNodeKey`, which hashes `nodeKeyString:attempt` with SHA-256 and
  maps each byte mod 26 to a letter).

### `IdentifierLookup` (database/identifier_lookup.js)

```
IdentifierLookup = {
    keyToId: Map<string, NodeIdentifier>,   // nodeKeyString → identifier
    idToKey: Map<string, NodeKeyString>,    // identifierString → nodeKey
}
```

Invariant: both maps are always inverses of each other (a strict bijection).  Mutations go through
`setIdentifierMapping`, which enforces this invariant by throwing `IdentifierLookupError` on any
conflict.

The lookup is serialised as a sorted array of `[NodeIdentifier, NodeKeyString]` pairs and stored
under the key `"identifiers_keys_map"` in the replica's global sublevel.

### `IdentifierResolver` (identifier_resolver.js)

Each top-level graph operation creates one resolver.  The resolver:

1. **Lazy-loads** the active lookup from `_computed.identifierLookup` on first use.
2. **Caches** every key↔id translation it performs in resolver-local maps, so the same key is never
   looked up twice.
3. **Allocates** new identifiers for keys that do not yet exist in the lookup, recording them in
   `pendingIdentifierMappings`.
4. Exposes `hasPendingAllocations` (boolean) and `applyPendingTo(activeLookup)` to merge pending
   entries into the shared lookup at commit time.

### `withIdentifierBatch` (graph_storage.js)

```
withIdentifierBatch(identifierResolver, fn):
    1. Build a BatchBuilder over the active schema storage.
    2. Await fn(batch)  ← runs OUTSIDE the mutex; produces the DB operations.
    3. Acquire withComputedStateMutex:
       a. If identifierResolver.hasPendingAllocations:
          - Read current _computed.identifierLookup.
          - Apply pending mappings onto it.
          - Append a rawPutOp for the updated identifiers_keys_map to the batch.
          - Flush the batch to LevelDB.
          - Replace _computed.identifierLookup with the new merged lookup.
       b. Otherwise: just flush the batch.
    4. Return the value from fn.
```

The mutex is per-replica (`computedStateIdentifier = rootDatabase.currentReplicaName()`), so
two replicas can commit concurrently without contention.

### `RootDatabaseClass._computed` (database/root_database.js)

```
ActiveReplicaComputed = {
    replicaName:       ReplicaName,           // 'x' | 'y'
    namespaceSublevel: SchemaSublevelType,
    globalSublevel:    GlobalSublevelType,
    schemaStorage:     SchemaStorage,
    identifierLookup:  IdentifierLookup,      // ← new field
}
```

All runtime state derived from the active replica lives exclusively here.  On startup,
`initializeActiveIdentifierLookup()` loads the persisted bijection into
`_computed.identifierLookup`.  On replica cutover (`setCurrentReplicaPointer`), `_computed` is
rebuilt from scratch including loading the new replica's lookup.

### Migration (migration_runner.js)

When a database with legacy key-addressed entries is opened:

1. The migration runner iterates every node in the legacy (key-addressed) replica.
2. For each node, `deterministicNodeIdentifierFromNodeKey` derives a stable identifier.
3. All node data (values, freshness, inputs, revdeps, counters, timestamps) is copied to the new
   (identifier-addressed) replica using the derived identifiers.
4. The identifier lookup is built from these (identifier, key) pairs and persisted.
5. The replica pointer is atomically switched to the new replica.

### `reconcile_identifier_lookup.js`

When merging a remote host's data during synchronisation, the remote host's identifier lookup may
conflict with the local lookup (same key → different identifier, or same identifier → different
key). `reconcileHostLookupWithTargetLookup` resolves these conflicts by making the remote lookup
yield to the local lookup for any conflicting assignments: remote entries that clash with local
assignments are overwritten to use the local identifier.
