# PR 1335: Switch IncrementalGraph persistence and migration to node identifiers

## Source studied

Pull request: <https://github.com/ottojung/volodyslav/pull/1335>

At the time of this review the PR is the draft branch `copilot/switch-to-unique-identifiers` against `master`. Its headline goal is to move IncrementalGraph durable storage away from semantic node keys and onto opaque node identifiers, while preserving migration behavior for existing replicas.

## High-level content

PR 1335 changes the identity boundary of the incremental graph.

Before this work, graph sublevels were conceptually keyed by the serialized semantic node key: the node head plus its argument values. That made storage easy to inspect, but it also meant that a user-facing key was the durable address of every value, freshness marker, input record, reverse-dependency record, counter, and timestamp.

After this work, normal graph storage is identifier-native:

- every node-data sublevel is keyed by `NodeIdentifier`;
- the semantic `NodeKeyString` is translated to a `NodeIdentifier` at the incremental-graph boundary;
- the durable database stores a global `identifiers_keys_map` record containing the bijection between identifiers and semantic keys;
- migration code knows how to read legacy key-addressed replicas and produce identifier-addressed replicas;
- graph storage helpers no longer need to understand node heads, node arguments, or serialized semantic-key structure.

Conceptually, this is a separation between **meaning** and **address**:

- `NodeKeyString` says what computation a node represents.
- `NodeIdentifier` says where that node's durable graph records live.
- `IdentifierLookup` is the only bridge between the two.

This is the PR's main architectural improvement. It narrows semantic-key knowledge to graph construction, pull/invalidation entry points, lookup/migration, and tests that intentionally inspect identity translation. The persistent graph core can operate on identifiers as opaque stable addresses.

## Important affected paths

### Identifier lookup and node identifiers

The PR adds the identifier model under `backend/src/generators/incremental_graph/database/`:

- `node_identifier.js` defines the opaque identifier type, conversion helpers, and validation boundary.
- `identifier_lookup.js` defines the in-memory/persisted bijection, allocation helpers, transaction overlays, serialization, and commit-time overlay publication.
- `reconcile_identifier_lookup.js` reconciles lookup data during sync/merge flows.
- `types.js`, `encoding.js`, and `index.js` export the new types and persistence codecs.

The significant low-level choice is that the lookup is represented as two maps:

```text
keyToId: NodeKeyString -> NodeIdentifier
idToKey: NodeIdentifier string -> NodeKeyString
```

The persisted form is a sorted array of `[NodeIdentifier, NodeKeyString]` pairs stored under `global/identifiers_keys_map`. Sorting by identifier gives deterministic fixture output and deterministic synchronization comparisons.

### Root database state

`backend/src/generators/incremental_graph/database/root_database.js` now treats the active replica's computed state as the holder of the active schema storage and the committed identifier lookup. On open, replica switch, migration, and reset, the active lookup is loaded from the active replica's global metadata.

Important consequences:

- the volatile lookup is a mirror of durable `identifiers_keys_map`;
- durable writes must happen before volatile lookup publication;
- active-replica changes must refresh the volatile lookup together with schema-storage handles;
- graph code can ask the root database for key-to-id and id-to-key lookups without opening graph sublevels by semantic key.

### Graph storage replacement

The old `graph_storage.js` is removed and replaced by `graph_state.js`.

`graph_state.js` provides:

- identifier-keyed storage accessors for values, freshness, inputs, reverse dependencies, counters, and timestamps;
- a read-your-writes batch wrapper around the typed sublevels;
- a transaction object containing a batch, an identifier overlay, and an in-transaction in-flight pull map;
- disk-first transaction commit: write node records and lookup metadata in one LevelDB batch, then publish the identifier overlay to memory.

The important design choice is that the transaction lookup is an **overlay**, not a full cloned lookup. New allocations are recorded in the overlay. Reads check the overlay first and then the committed base. Commit serializes base plus overlay for disk and only mutates the base after the durable batch succeeds.

### Pull, recompute, and invalidation

The pull and recompute path now resolves concrete nodes into `ResolvedConcreteNode` objects containing both semantic keys and identifiers:

- semantic keys are still used to find compiled node definitions and to call computors;
- identifiers are used for all persistent graph records;
- nested pulls reuse the caller's transaction instead of creating a second transaction;
- the transaction's `inFlight` map deduplicates repeated pulls of the same node inside a transaction.

Invalidation similarly resolves the target semantic node to its identifier before writing freshness/input state and propagating stale status through identifier-keyed reverse dependencies.

### Migration

The migration runner and migration storage are made aware of both worlds:

- legacy replicas may contain graph records addressed by semantic key strings;
- migrated replicas contain graph records addressed by node identifiers;
- deterministic migration identifiers are derived from legacy node keys;
- collision handling is deterministic by retrying with an attempt index;
- migration materializes `identifiers_keys_map` and rewrites graph sublevels to identifier keys.

This is why PR 1335 has many fixture changes: path names in the rendered mock remote change from readable node names such as `all_events` or `config` to opaque identifiers such as `athsapppb`, and the fixture gains `global/identifiers_keys_map`.

### Synchronization and unification

Sync/merge/unification code is updated so database-to-database and database-to-filesystem flows compare identifier-addressed records and preserve/reconcile lookup metadata. The lookup is global metadata, while node state remains in typed graph sublevels.

### Tests

The PR adds and expands tests for:

- identifier allocation and lookup behavior;
- key/id bijection correctness;
- volatile-to-durable lookup consistency;
- migration runner behavior for legacy and migrated replicas;
- reverse-dependency ordering using identifiers;
- sync merge and unification behavior with identifier-addressed records;
- concurrency behavior around current transaction locking.

The tests are important because this PR changes a persisted format. The populated migration fixture is especially important: it proves that an existing rendered remote can be migrated into exactly the expected identifier-addressed shape.

## Low-level choices worth preserving

### Identifier-native storage must stay identifier-native

The storage layer should not regain knowledge of node heads, argument arrays, serialized node-key grammar, or semantic-key migration rules. If a helper in the graph storage core needs to parse a node key, the abstraction has leaked.

The intended boundary is:

1. public graph operation receives node name and bindings;
2. graph code serializes/looks up/allocates exactly once at the boundary;
3. persistent graph operations use only `NodeIdentifier` values thereafter.

### The lookup overlay is the right transaction unit

Using an overlay for newly allocated mappings avoids cloning a large lookup for every graph operation. It also makes the commit boundary explicit: only overlay entries are candidates for merge against the latest committed lookup.

This overlay is the only part of the transaction that needs to be treated like a mergeable delta. The ordinary node-data batch should remain ordinary batch operations, because per-node locking should make conflicting pull writes unrepresentable rather than requiring a full logical-diff engine.

### Disk-first publication is mandatory

If a transaction creates identifiers, the durable batch must include both node-data writes and the serialized lookup update. The volatile lookup can only be advanced after that batch succeeds. Otherwise inspection or later graph operations could observe an identifier mapping that cannot be reconstructed after restart.

### Migration identifiers are deterministic

Legacy migration must be repeatable. Deriving migration identifiers from the node key plus an attempt counter gives stable output while still allowing deterministic collision resolution. This is preferable to using runtime randomness during migration, because fixture output and recovery behavior need to be reproducible.

### Reverse-dependency lists stay sorted

Identifier-addressed reverse-dependency lists are maintained in sorted identifier order. This keeps durable output deterministic and avoids order-only fixture churn.

## Relationship to the locking redesign

PR 1335 solves the identity and migration problem, but it currently does so with a broad transaction mutex around the whole transaction body. That mutex is correct for volatile/durable consistency, but it serializes too much work: computor execution, dependency traversal, identifier allocation, and commit all happen under the same computed-state lock.

The follow-up design in `docs/design.md` should therefore be read as a simplification and narrowing of the locking plan for PR 1335's implementation. It keeps the identifier overlay and disk-first commit model, but replaces the over-broad transaction-body lock with:

- the existing graph activity mode lock;
- per-node pull locks held until transaction completion;
- synchronous in-process identifier reservation for newly generated identifiers;
- a short commit mutex that covers only durable commit and volatile publication.
