# PR #1335 — Switch IncrementalGraph persistence and migration to node identifiers

## High-level goal
PR #1335 moves IncrementalGraph persistence away from storing graph payloads keyed by serialized `NodeKey` values and toward a model centered on stable `NodeIdentifier` values. The objective is to decouple persistence identity from mutable key shape and make migration/cutover between replicas (`x`/`y`) explicit and safer.

## Conceptual changes

1. **Identifier-native persistence path**
   - The storage and graph layers are refactored so runtime graph operations resolve keys to identifiers and persist data via identifiers.
   - Identifier lookup data is persisted in replica-local global state.

2. **Replica-aware identifier lookup ownership**
   - The active lookup cache now follows `_meta/current_replica`.
   - On replica switch, in-memory lookup state is refreshed from the newly active replica.

3. **Migration choreography around replica cutover**
   - Migration writes transformed state into inactive replica first.
   - Replica pointer switch (`setCurrentReplicaPointer`) marks cutover.
   - Tests enforce ordering expectations around pre/post commit and cutover.

4. **Test and docs alignment**
   - Database, migration, unification, and sync tests were expanded/adjusted to include `_meta/current_replica` semantics and identifier lookup behavior.
   - Design/docs were updated to describe `/${current_replica}/global/identifiers_keys_map` and boot/cutover assumptions.

## Low-level implementation choices visible from the branch

- `RootDatabaseClass` caches current replica in memory and exposes:
  - `currentReplicaName()`, `otherReplicaName()`
  - `schemaStorageForReplica(name)` for dual-replica migration access
  - `setCurrentReplicaPointer(name)` to persist `_meta/current_replica`, update cache, and refresh identifier lookup
- Identifier translation helpers (`nodeKeyToId`, `nodeIdToKey`) are now rooted in loaded lookup state rather than ad-hoc key serialization in graph-state file layout.
- Boot behavior ensures `_meta/current_replica` defaults to `x` for fresh/legacy DBs and rejects invalid values.

## Trade-offs in PR #1335

- **Pros**
  - Clearer identity model (`NodeIdentifier` as canonical persisted identity).
  - Better migration structure through explicit replica staging and cutover.
  - More direct assertions in tests around replica pointer semantics.

- **Cons / residual risk**
  - Replica switch currently persists pointer before full readiness validation of new active runtime state (addressed in review feedback and follow-up docs below).
