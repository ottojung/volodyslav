# PR #1335 — Switch IncrementalGraph persistence and migration to node identifiers

## High-level objective
PR #1335 changes IncrementalGraph persistence to be **identifier-native** instead of persisting semantic node keys directly. The central conceptual shift is:

- semantic node keys remain the runtime/domain concept;
- persistent storage uses opaque `NodeIdentifier` values;
- semantic-key ↔ identifier mapping is handled in dedicated lookup structures and migration/sync logic.

This decouples durable storage from semantic key string representations and prepares the graph store for stronger persistence invariants.

## Conceptual architecture introduced

### 1) Identifier-native storage boundary
The PR establishes `graph_storage` as an identifier-only persistence layer. Semantic key materialization/translation is moved up to IncrementalGraph orchestration/runtime paths.

### 2) Identifier lookup lifecycle
A durable `identifiers_keys_map` lookup is merged and persisted as part of replica synchronization and write operations. Runtime operations resolve or allocate identifiers and queue lookup persistence with the same write batch.

### 3) Migration + sync responsibilities
Merge/sync code now has to keep value/timestamp/freshness/revdeps data and identifier lookup state coherent across replicas.

## Low-level implementation choices in the PR

- Introduced/expanded identifier lookup helpers in the IncrementalGraph database layer (allocation, serialization, reverse lookup, merge).
- Added runtime resolver behavior (`identifier_resolver`) so pull/recompute paths can:
  - resolve existing semantic key identifiers,
  - allocate identifiers when missing,
  - persist lookup updates together with graph writes.
- Adjusted pull/recompute codepaths to operate with identifier-focused APIs.
- Updated sync/merge path to merge host and target identifier maps when graph changes are applied.
- Added/updated tests for migration and identifier-based operations, including compatibility around legacy data and mixed lookup states.

## Design tradeoffs visible in the PR

- **Pros:** clearer persistence invariants, easier future key evolution, cleaner storage abstraction.
- **Costs:** more moving parts in runtime translation, sync correctness now depends on lookup flush semantics, and identifier/key ambiguity becomes a correctness risk in pull entrypoints.

## Review-sensitive areas
The PR’s complexity clusters around:

1. synchronization flush ordering (`pendingOps` and batch boundaries);
2. collision retry behavior when deterministic fallback identifiers are generated;
3. semantic-key vs persisted-identifier ambiguity in pull internals.

These areas are exactly where review feedback later concentrated.
