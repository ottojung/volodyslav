# PR #1335 – Switch IncrementalGraph persistence and migration to node identifiers

## High-level intent

PR #1335 transitions IncrementalGraph storage from semantic node keys (head + args) to stable `NodeIdentifier` strings as the persisted key space. The core design goal is to keep semantic key ↔ identifier translation at the boundary, then run all storage-level operations with identifier-native keys.

Conceptually, this delivers:

- a stable persisted identity for nodes that is decoupled from key-shape details,
- fewer repeated key/identifier lookups during graph operations,
- cleaner migration boundaries (legacy data rewritten once into identifier form), and
- clearer layering: graph runtime semantics above, storage implementation below.

## Architectural choices

### 1) Identifier-native storage core

`backend/src/generators/incremental_graph/graph_storage.js` is structured around persisted identifiers, not semantic key composition. This aligns with review guidance that storage should not carry NodeKey semantics.

### 2) Explicit lookup map in replica-global state

A lookup map (`identifiers_keys_map`) is maintained in replica-global metadata and loaded into in-memory lookup structures in `root_database.js`. This map is the controlled translation boundary for:

- external/semantic key addressing at API edges,
- internal identifier-addressed persistence for values/freshness/inputs/revdeps/counters/timestamps.

### 3) Migration strategy

Legacy snapshots that were keyed by semantic key strings are migrated into identifier-keyed forms. Migration is done as a data-shape translation step, preserving graph meaning while changing storage addressing.

### 4) Replica-aware runtime

The root database maintains `x`/`y` replica sublevels with an active pointer in `_meta/current_replica`. Runtime callers use active-replica accessors while migration/sync paths can access explicit replica storages.

## Low-level details worth noting

- Storage records such as inputs/revdeps are persisted by `NodeIdentifier` values.
- Type and JSDoc surface area was expanded/refined to keep identifier-vs-key distinctions explicit.
- Tests were reworked to preserve migration and storage behavior checks after key-space changes.
- Follow-up commits after #1335 further stabilized replica cutover and migration ordering behavior.

## Net effect

PR #1335 is a data-model pivot: persisted graph topology and node state are now fundamentally identifier-addressed. Semantic keys remain important, but only at translation boundaries, which improves determinism and reduces storage-layer coupling to semantic key encoding.
