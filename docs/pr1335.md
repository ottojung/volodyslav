# PR #1335 Study Notes

## Scope and intent
PR [#1335](https://github.com/ottojung/volodyslav/pull/1335) is a broad refactor of IncrementalGraph persistence and migration mechanics around one central conceptual shift:

- **Persist data by `NodeIdentifier` everywhere in storage-facing layers**.
- Move any `NodeKey` ↔ `NodeIdentifier` translation to graph/runtime boundaries, rather than mixing key semantics into low-level storage code.

At a high level, this reduces ambiguity in persistence behavior and makes migration/synchronization logic operate on one canonical identity representation.

## High-level conceptual changes

1. **Identifier-native storage model**
   - Storage internals (values/freshness/inputs/revdeps/counters/timestamps/global metadata) now treat identifiers as the first-class persisted keys.
   - The identifier lookup map (`identifiers_keys_map`) becomes the single bridge between logical keys and persisted identifiers.

2. **Migration/sync flow alignment**
   - Replica migration and sync paths are adapted to work with identifier-native records.
   - Merge/unification/reset flows are updated so they preserve identifier-addressed invariants.

3. **IncrementalGraph runtime boundary clarification**
   - Runtime operations (recompute/invalidate/pull/inspection/instantiation, etc.) keep user-facing key semantics.
   - Translation is performed at call boundaries, then storage interactions remain identifier-native.

4. **Test fixture and regression suite updates**
   - Existing tests were updated to reflect new persistence shapes.
   - Rendered fixture snapshots now include identifier-key map and identifier-addressed storage artifacts.

## Low-level, concrete choices visible in the PR

- `backend/src/generators/incremental_graph/database/root_database.js`
  - Keeps active/inactive replica routing, identifier generation, and identifier lookup loading within root database boundaries.
- `backend/src/generators/incremental_graph/graph_storage.js`
  - Evolved toward a pure storage module centered on identifiers.
- `backend/src/generators/incremental_graph/database/identifier_lookup.js`
  - Lookup operations are explicit and centralized.
- Migration and synchronization code (`migration_runner.js`, `sync_merge.js`, `synchronize*.js`, unification modules)
  - Updated to use identifier-based reads/writes consistently.
- Extensive test updates in `backend/tests/`
  - Assertions now exercise identifier-addressed behavior and migration/snapshot correctness under the new model.

## Architectural implications

- **Positive**: Reduced representational drift across modules and lower risk of key/id mismatch bugs in storage.
- **Tradeoff**: The active-replica + active-lookup caches must stay perfectly synchronized after pointer cutovers, because runtime APIs read from cached state rather than re-reading metadata for each call.
- **Operational sensitivity**: Checkpoint and migration sequencing now depends more strongly on in-process cache coherence after replica switching.
