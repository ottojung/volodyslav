# PR #1335 — Switch IncrementalGraph persistence and migration to node identifiers

## High-level goal

PR [#1335](https://github.com/ottojung/volodyslav/pull/1335) moves IncrementalGraph persistence away from semantic node keys and toward opaque `NodeIdentifier` values as the primary storage key.

Conceptually, this means:

1. Runtime graph storage (`values`, `freshness`, `inputs`, `revdeps`, `counters`, `timestamps`) is keyed by stable identifiers.
2. Semantic keys (`head + args`-derived node keys) are translated at operation boundaries instead of being threaded through lower-level storage APIs.
3. A persisted global bijection (`identifiers_keys_map`) tracks `NodeIdentifier <-> NodeKey` so persisted data remains inspectable and migratable.

## Architectural shape

### 1) Identifier lookup as first-class persisted metadata

The PR introduces/expands `identifier_lookup` behavior:

- Global metadata key: `identifiers_keys_map`.
- In-memory two-way lookup:
  - `keyToId: Map<string, NodeIdentifier>`
  - `idToKey: Map<string, NodeIdentifier>`
- Deterministic serialization sorted by identifier for stable persistence and diffs.
- Allocation path with collision handling and explicit error types.

This centralizes the semantic-key translation logic and keeps storage tables identifier-native.

### 2) RootDatabase tracks active-replica lookup

`RootDatabase` now caches the active replica’s lookup and exposes methods to:

- clone active lookup snapshots for operation-local work,
- replace cached active lookup after successful persistence,
- perform lookup queries (`nodeKeyToId`, `nodeIdToKey`),
- initialize/reload lookup when switching replicas.

### 3) Per-operation resolver for IncrementalGraph

`identifier_resolver` holds a mutable per-operation snapshot:

- resolve existing identifier,
- allocate if needed,
- cache reverse mapping for repeated access,
- append persistence writes into the batch,
- publish successful state back to the active `RootDatabase` cache.

This is the key separation boundary: **IncrementalGraph runtime resolves keys; graph storage remains identifier-only**.

## Migration implications

The PR also updates migration pathways so legacy snapshots keyed by older key forms can be converted into identifier-keyed records and lookup metadata can be preserved/restored.

## Low-level choices worth calling out

1. **Bijection strictness**: duplicate ids or duplicate semantic keys are treated as integrity errors.
2. **Deterministic ordering**: serialized map entries are sorted by identifier to avoid nondeterministic persisted output.
3. **Operation-local mutability**: each pull/invalidate can mutate its own snapshot, then persist/commit as a unit.
4. **Compatibility path for test doubles**: resolver has a fallback `WeakMap` lookup path when a root-database double does not implement the newer lookup API.

## Known review pressure points

The review thread specifically focuses on concurrency semantics around lookup persistence and memory/persistence consistency under failures. Those concerns are addressed in follow-up notes and implementation planning documents in this branch.
