# PR #1335 Study Notes

## Title and intent
PR [#1335](https://github.com/ottojung/volodyslav/pull/1335) is titled **"Switch IncrementalGraph persistence and migration to node identifiers"**.

Its architectural goal is to complete a storage-model transition:
- Runtime/persistent sublevels (`values`, `freshness`, `inputs`, `revdeps`, `counters`, `timestamps`) should be keyed by opaque **node identifiers**.
- Semantic keys (`NodeKey`, serialized JSON) remain the API-level addressing format and are translated through an identifier lookup.
- Migration must work for both pre-identifier and identifier-native replicas.

## High-level conceptual change
The PR introduces and threads a global map, `identifiers_keys_map`, that stores a bijection:
- identifier -> semantic node key
- and (in memory) semantic node key -> identifier

This lets runtime logic keep semantic-level APIs while persistence remains identifier-native.

## Main low-level choices in the PR
- Introduced identifier lookup utilities (`identifier_lookup.js`) with:
  - deterministic identifier derivation,
  - collision-aware allocation,
  - validation of bijection invariants,
  - serialization/deserialization for persistence.
- Refactored incremental graph persistence layer toward identifier-centric behavior.
- Updated migration runner to create a key-translation plan that maps migration decisions from semantic keys to output identifiers.
- Added/updated tests across migration, persistence, database, and resolver areas.

## Data model implications
- `global/version` remains migration/version control metadata.
- `global/identifiers_keys_map` becomes mandatory metadata once identifier-native mode is present.
- Runtime reads/writes must preserve consistency between per-sublevel records and this lookup.

## Why this PR is important
The change is foundational: once storage keys become opaque identifiers, all compatibility boundaries (migration, pull, sync, revdeps reconstruction, inspection) depend on correct key translation and identifier-map persistence.
