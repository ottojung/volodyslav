# PR #1335 Study Notes

## High-level intent
PR #1335 (“Switch IncrementalGraph persistence and migration to node identifiers”) changes the incremental-graph persistence model from semantic NodeKey-addressed storage to opaque NodeIdentifier-addressed storage. The conceptual goal is to make persistence stable under schema/key-shape changes while keeping semantic meaning in a dedicated identifier lookup map.

## Conceptual architecture changes
- Node identity in persisted stores is now an opaque 9-letter identifier.
- Semantic node keys are still available, but through a lookup (`global/identifiers_keys_map`) rather than as storage keys.
- Migration/cutover flows rely on replica swap (`x`/`y`) so state transitions remain atomic from reader perspective.
- Render/scan and synchronization paths were updated to treat identifier-addressed graph state as first-class.

## Low-level choices visible in this branch
- Replica pointer state is cached in root database runtime state and switched through `setCurrentReplicaPointer`.
- Sync merge writes operate against inactive replica, then pointer cutover occurs only when changes exist.
- Snapshot reset import also uses inactive replica + pointer switch.
- Encoding/decoding maps DB keys to filesystem paths with `_meta` vs replica-depth path handling.
- Pull/recompute paths use identifier resolver to map between persisted IDs and semantic node definitions.

## Risk areas
- Any path that writes node/revdeps data without updating identifier lookup can create “dangling” IDs.
- Any decoder that rejects legacy path layouts can break upgrade from old on-disk snapshots.
- Any API that claims `NodeIdentifier` input but deserializes as NodeKey is semantically inconsistent.
